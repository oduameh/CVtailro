CVtailro Technical Documentation

Comprehensive reference for the AI resume tailoring platform. Covers architecture, pipeline, APIs, deployment, and troubleshooting.

Architecture Overview

CVtailro is a production AI resume tailoring SaaS built with Python/Flask. Users sign in, upload a PDF resume + paste a job description, and receive an AI-optimized resume in multiple templates.

Business Model

Admin (you) provides the LLM API key. Users get the service for free. Admin pays API costs. Supports OpenRouter and NVIDIA NIM as LLM backends.

End-to-End Data Flow

User uploads PDF + pastes job description → load_resume() extracts text from PDF → Stages 1+2 PARALLEL: Job Intelligence Agent → JobAnalysis (structured job data) Resume Parser Agent → ResumeData (structured resume data) → Stage 3: Gap Analysis (pure Python, instant) → GapReport (match score, missing keywords, cosine similarity) → Stage 4: Bullet Optimiser (parallel per role if 3+ roles) → OptimisedBullets (rewritten bullets with keyword injection) → Stages 5+6 PARALLEL: Resume Optimiser → ATSResume (unified ATS+recruiter markdown) Talking Points → list[TalkingPoint] (interview prep) → Post-pipeline: Resume quality scoring, email templates, keyword density, cover letter → Output: PDFs (all 8 templates) + DOCX + Match Report JSON + Talking Points MD → Storage: DB update + R2 upload + JobFile records → User sees: Match score, keyword analysis, download buttons, interview prep

6-Stage Pipeline

Orchestrated by app/services/pipeline.py. Runs in a background thread with concurrency limits.

Concurrency Controls

  • MAX_CONCURRENT_PIPELINES = 5 — semaphore limits active jobs
  • MAX_QUEUE_DEPTH = 50 — rejects new jobs when queue is full (503)
  • JOB_TTL_SECONDS = 900 — completed jobs cleaned from memory after 15 min
  • Progress events pushed to per-job queue.Queue(), consumed by SSE endpoint

LLM Call Flow (BaseAgent._call_llm_api)

  • Routes to OpenRouter (openrouter.ai/api/v1/chat/completions) or NIM (integrate.api.nvidia.com/v1/chat/completions) based on provider
  • NIM: max_tokens capped at 4096 (NIM API hard limit)
  • Retry logic: up to 5 attempts with exponential backoff; permanent errors (401, 402, 410) skip retries
  • NIM error parsing handles both OpenAI format (error.message) and NIM format (detail/title)
  • Reasoning models (e.g., DeepSeek R1): checks reasoning_content field if content is empty

Data Models (Pydantic)

Defined in models.py. Key models:

  • JobAnalysis — job_title, company, required_skills, preferred_skills, tools, seniority, responsibilities
  • ResumeData — name, contact, summary, roles (with bullets), education, certifications, global_skills
  • GapReport — match_score, cosine_similarity, missing_keywords, keyword_frequency, section_scores
  • OptimisedBullets — mode_used, bullets (with original/optimised text, keywords injected, fabrication flag)
  • ATSResume — markdown_content, ats_checks, keyword_density, suggested_title
  • MatchReport — overall_match_score, cosine_similarity, missing_keywords, ats_checks
  • TalkingPoint — topic, bullet_points, source_experience

CSRF Cookie (Double-Submit Pattern)

After every response: if csrf_token cookie is missing OR session has no CSRF token, generates a new token via generate_csrf() and sets a readable cookie. Frontend reads this cookie and sends it as X-CSRFToken header on POST/PUT/DELETE requests.

OpenRouter Provider

  • API URL: https://openrouter.ai/api/v1/chat/completions
  • Env var: OPENROUTER_API_KEY (always overrides DB)
  • Default model: openai/gpt-4o-mini
  • Models: 23 curated models (free + paid), hardcoded in config.py
  • Max tokens: Uses agent's full AGENT_MAX_TOKENS (up to 8192)

NVIDIA NIM Provider

  • API URL: https://integrate.api.nvidia.com/v1/chat/completions
  • Env var: NVIDIA_NIM_API_KEY (always overrides DB)
  • Default model: First available model from dynamic discovery
  • Models: Dynamically fetched from /v1/models for the account (cached 5 min)
  • Max tokens: Capped at 4096 (NIM API hard limit)
  • Free tier: Only certain models available per account (404 for unavailable models)

Provider Switching

  1. Admin panel → Configuration → Active Provider radio
  2. Save Configuration → active_provider saved to DB
  3. Model dropdown refreshes automatically (fetches /api/models?provider=...)
  4. Default model validated against provider's model list
  5. All API endpoints (/api/tailor, /api/boost-bullet, /api/batch-tailor) use the active provider's key and URL

Main App (index.html)

Page Load

  1. loadSettings() → fetches /api/status and /api/models
  2. Model dropdown populated with Free/Premium groups
  3. Selected model persisted in localStorage['cvtailro_model']
  4. Dark mode from localStorage['cvtailro-theme'] or system preference

Tailoring Flow

  1. User uploads resume (PDF/MD/TXT) + pastes job description
  2. startTailoring() → FormData POST to /api/tailor
  3. SSE connection to /api/progress/<job_id> for real-time progress
  4. On complete: loadResults() renders score, keywords, template cards
  5. Download via safeDownload() (fetch + blob + <a download>)

CSRF Flow

Global fetch() interceptor adds X-CSRFToken header to all non-GET requests, reading the token from csrf_token cookie.

Troubleshooting

Healthcheck fails on deploy (503 / replicas never healthy)

  • Check Railway build logs for pip install errors (missing system dep?)
  • Check if FLASK_SECRET_KEY is set — production config raises RuntimeError without it
  • Check if a new dependency requires compilation (needs gcc in Docker)
  • The /api/health endpoint always returns 200; if it's failing, the app isn't starting at all

CSRF token missing (POST returns 500)

  • User's browser has a stale session cookie from a previous deployment
  • Fix: the after_request handler regenerates CSRF when csrf_token is missing from the session
  • Worst case: user clears cookies and reloads

NVIDIA NIM: "Function not found for account" (404)

  • The NIM free tier only grants access to certain models per account
  • Fix: the model list is now dynamically fetched from /v1/models for your account
  • Admin panel → Test Active Key will show which models are available

NVIDIA NIM: "Internal server error" (500) on all models

  • Most likely cause: max_tokens exceeds NIM's 4096 limit
  • Fix: base_agent.py caps max_tokens at 4096 for NIM
  • Also check: API key validity (Test Active Key in admin panel)

NVIDIA NIM: model deprecated (410)

  • NVIDIA removes models from the free catalog periodically
  • Error is now non-retryable (skips all 5 retry attempts)
  • Fix: select a different model from the dropdown, or switch to OpenRouter

API keys not loading from Railway env vars

  • OPENROUTER_API_KEY and NVIDIA_NIM_API_KEY env vars always override DB values
  • This ensures rotated keys on Railway are picked up immediately
  • Check Railway dashboard → Variables to verify the env vars are set

Model dropdown shows wrong provider's models

  • After switching provider in admin, click Save Configuration
  • The model dropdown refreshes automatically after save
  • Users need to reload the main page to get the new models

Downloads fail (files not found)

  • Three-tier download: local files → R2 → DB regeneration
  • Local files expire after ~5 minutes (cleaned up by background thread)
  • R2 files persist if R2 is configured; otherwise relies on DB regeneration
  • DB regeneration rebuilds PDFs from stored markdown on demand

Pipeline stuck / queue full (503)

  • Max 5 concurrent pipelines, queue of 50
  • Check admin panel → System tab for active pipelines and queue depth
  • Long-running pipelines (>5 min) usually indicate LLM timeout
  • Fix: restart the Railway service to clear stuck semaphores

OpenRouter: rate limited (429)

  • Pipeline waits Retry-After seconds (max 30s) then retries
  • If persistent: reduce concurrent pipelines or switch to a less popular model
  • Free models are more likely to be rate-limited

Database connection errors

  • Connection pool: 20 + 30 overflow, 10s timeout, 300s recycle, pre-ping enabled
  • "There is already a transaction in progress" — stale connections after deploy, self-resolves via pool_pre_ping
  • Check admin panel → Diagnostics tab for DB health
